C4 · 1

Contexto

Quién usa Perito y de qué sistemas externos depende. Una sola caja: el sistema completo.

Perito hace la admisión (extraer → verificar → cobertura → fraude → HITL) y delega el razonamiento a Claude y la observabilidad a Langfuse. No hace pagos, enrutamiento al ajustador ni integración con el core — eso es Won't.

C4 · 2

Contenedores

Las piezas desplegables. Un monolito modular, server-rendered (ADR-001 · ADR-002).

Una sola caja desplegable: el navegador llama al Workbench por HTTPS, que consume el backend por REST server-side (autorización siempre en el servidor). Ámbar: pgvector existe como esquema pero el RAG de pólizas no está cableado — la cita de cláusula se logra por policy_lookup determinístico.

C4 · 3

Componentes — el pipeline agéntico real

Dentro del backend: los nodos C0–C11 tal como se construyeron. El orquestador (C7) es dueño de la terminación (P4).

El aviso entra por C0/C1, el orquestador C7 corre C2→C6 (con el loop reflexivo extractor↔verificador acotado), el analista redacta el resumen, y todo aterriza en el Workbench donde el humano firma — ningún camino alcanza un estado terminal sin aprobado_por (P1). Puntos verdes = emergentes del build (triage, entity resolution, cross-claim, analista, poller). Ámbar: RAG vectorial y copiloto siguen como stub/mock — declarados, no escondidos (P7).
